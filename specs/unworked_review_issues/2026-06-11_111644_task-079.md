2. [x] **code-quality-reviewer** | `test/integ/digest_client.hpp:771` | correctness
   make_cnonce produces only 8 distinct hex chars (nibbles) per 64-bit word instead of 16. The loop runs 8 times and each iteration takes only the low nibble of one byte: `digits[(hi >> (i * 8)) & 0xF]`. This masks out the high nibble of every byte, halving the entropy space from 64 bits to 32 bits. The high nibble of each byte is silently discarded. For a test-only cnonce the security impact is negligible, but the function comment says '16-hex-char' yet only generates 8 meaningfully distinct hex chars per word.
   *Recommendation:* Either extract both nibbles per byte (shifting by i*8+4 for the high nibble as well), or simplify to a `std::uniform_int_distribution<uint8_t>` loop over 8 bytes producing 16 hex chars. The simplest fix: `out[2*i] = digits[(byte >> 4) & 0xF]; out[2*i+1] = digits[byte & 0xF];` for each of 8 bytes drawn from the RNG.
   *Worked 2026-07-07 (chore/unworked-major-issues):* Already resolved by commit 52a06e0: make_cnonce now emits 32 hex chars from 16 random bytes via detail::to_hex_lower (both nibbles of every byte).

30. [x] **security-reviewer** | `test/integ/digest_client.hpp:771` | cryptographic-failures
   make_cnonce() extracts nibbles from the two 64-bit MT outputs using `(hi >> (i*8)) & 0xF` for i in [0,7]. Each iteration shifts by i*8 bits (0, 8, 16, … 56) and masks the low nibble. This recovers only the low 4 bits of every byte — one nibble per byte position. The high nibble of each byte is discarded, halving the effective entropy of the 16-hex-char output to at most 32 bits (one nibble per byte × 8 bytes × 2 values = 64 bits total, but the same nibble position from each byte is not independent under MT). The result is that the output looks like a 16-hex-char string but is generated from far fewer distinct outputs than a full random 64-bit integer would provide.
   *Recommendation:* Use `digits[(hi >> (i*4)) & 0xF]` stepping by 4 bits rather than 8, so all 16 nibbles of the 64-bit hi value are consumed. Example fix:
  for (int i = 0; i < 16; ++i) {
      uint64_t val = (i < 8) ? hi : lo;
      out[i] = digits[(val >> ((i % 8) * 4)) & 0xF];
  }
   *Worked 2026-07-12 (chore/unworked-minor-issues):* Already resolved in the current tree — The nibble-extraction bug this finding describes (`(hi >> (i*8)) & 0xF` discarding the high nibble of every byte) no longer exists: current make_cnonce (test/integ/digest_client.hpp:764-773) memcpy's both 64-bit words into a 16-byte buffer and calls detail::to_hex_lower(buf, 16), which emits both nibbles of every byte -- the same fix already recorded against major finding #2 (commit 52a06e0).
