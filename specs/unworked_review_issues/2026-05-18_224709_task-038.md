22. [ ] **security-reviewer** | `.github/workflows/verify-build.yml:482` | data-integrity
   libmicrohttpd-1.0.3.tar.gz is fetched with `curl https://libhttpserver.s3.amazonaws.com/...` (plain HTTP, not HTTPS) before the SHA-256 integrity check. A network-level attacker (MitM on the S3 endpoint) could substitute a malicious tarball; curl would accept it and only the sha256sum check would stand between the attacker and arbitrary code execution in the CI runner. The same pattern is repeated for the no-dauth (line 544), flag-invariance-off (line 554), Windows (line 605), and ARM cross-compile (line 623) download steps. SHA-256 pinning limits the risk but HTTP delivery means the digest can never authenticate the transport, only the content — a valid defence against a CDN or S3 misconfiguration but not against a targeted substitution attack where the same hash-matching payload is served.
   *Recommendation:* Replace the `http://` (or unspecified S3 bucket HTTP redirect) URL with `https://` in every `curl` invocation that fetches libmicrohttpd. Since the bucket URL already appears to resolve to S3 over HTTPS in practice, the change is low-risk: `curl https://s3.amazonaws.com/libhttpserver/libmicrohttpd_releases/libmicrohttpd-1.0.3.tar.gz`. Also consider adding `curl --fail --location --proto '=https'` flags to prevent silent redirect to a non-TLS endpoint.
