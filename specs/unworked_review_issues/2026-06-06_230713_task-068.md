1. [x] **security-reviewer** | `src/httpserver/detail/connection_state.hpp:135` | sensitive-data-handling
   Heap overflow blocks from monotonic_buffer_resource are freed but NOT zeroed by reset_arena(). When a request's arena usage exceeds ARENA_INITIAL_BYTES (8 KiB) — e.g., a request with a large body, many headers, or a large credential string — MHD silently allocates additional blocks from the upstream (heap) allocator. reset_arena() calls arena_.release() which reclaims those blocks back to the heap free list, but does not zero them. The credential bytes in those overflow blocks are then observable to any future allocation from the same process heap (e.g., via another connection on the same thread pool). The comments in the header acknowledge this gap and explicitly accept it for this task, but the residue is a genuine CWE-226 window that persists in production.
   *Recommendation:* Track this as a required follow-up security task. To close the gap, consider: (a) a custom zero-on-deallocate upstream memory_resource adapter that wraps new_delete_resource and calls secure_zero before every deallocation, or (b) a hand-rolled arena that keeps a list of all overflow blocks and zeroes+frees them on release. The accepted-residue comment in connection_state.hpp (line 144-148) should reference a concrete tracking issue number so it cannot silently languish.
   *Worked 2026-07-07 (chore/unworked-major-issues):* Tracked as TASK-095 (specs/tasks/M7-v2-cleanup/TASK-095.md, Backlog, with the zero-on-deallocate adapter vs hand-rolled arena options and an overflow sentinel test); the accepted-residue comment in connection_state.hpp now references TASK-095 concretely.

27. [ ] **security-reviewer** | `src/httpserver/detail/secure_zero.hpp:116` | cryptographic-failures
   Portable fallback: the volatile pointer is correctly typed as `volatile unsigned char*`, making each individual array element write a volatile access (visible side effect per [intro.execution]). However, the C++ standard only guarantees that volatile accesses are not reordered relative to each other, not that they are not optimized away relative to non-volatile memory. Under C++17/20 with whole-program LTO and a compiler that can prove no subsequent volatile read exists (which is the case on the no-keep-alive tear-down path), a conforming optimizer could still elide the stores. The inline-asm memory clobber (`__asm__ __volatile__("": :"r"(p):"memory")`) that follows is the actual DCE barrier on GCC/Clang and is correct. On MSVC (non-Windows paths excluded by the `_WIN32` branch but relevant if the code is ever ported under Clang-CL on Windows), there is no equivalent asm fence. The comment in the header correctly notes macOS takes the fallback; macOS Clang is the primary user of this path in practice and the asm clobber covers it.
   *Recommendation:* Document explicitly in the header that the volatile loop alone is not sufficient and that the asm clobber is the load-bearing barrier. Consider adding `#if !defined(__GNUC__) && !defined(__clang__) && !defined(_WIN32)` with a `#pragma message` or a `static_assert(false, ...)` to catch future ports to non-GCC/Clang non-Windows compilers where neither the asm barrier nor RtlSecureZeroMemory is available.
