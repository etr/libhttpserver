34. [ ] **performance-reviewer** | `test/integ/threadsafety_stress.cpp:736` | race-condition
   The latency gate baseline is computed from samples_ns (insertion-order vector) but samples_ns is populated under samples_mtx by four concurrent writer threads. The warmup quarter is therefore interleaved across all four threads — it is NOT the first quarter of low-cardinality operations; it is the first N/4 samples in whatever order the mutex was won. On a multicore host threads 1-3 may have already registered hundreds of high-cardinality routes before thread 0's first batch lands, so the 'warmup_median' can already reflect high-cardinality latency. This makes the 10× gate trivially pass: both baseline and p99 are drawn from similar distributions, and the spike signal is diluted. Additionally, the warmup vector is re-locked separately from sorted (two separate lock_guard scopes) so in theory warmup and sorted could diverge if more samples arrive between the two locks — though in practice the writers have joined before this code runs, so the second lock is a benign double-copy rather than a true race.
