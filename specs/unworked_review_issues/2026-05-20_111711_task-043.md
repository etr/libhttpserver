4. [x] **test-quality-reviewer** | `scripts/check-doxygen.sh:80` | missing-test
   The warning-detection grep pattern is '^|(: )(warning|error):'. Doxygen occasionally emits warnings in the form 'filename:line: warning: ...' with the colon immediately after the line number and NO space before 'warning:'. The pattern requires either a line-start (^) or '(: )' (colon-space) preceding the keyword. A warning of the form '/path/to/file.hpp:42: warning: ...' has no space before the keyword, so the colon-space branch '(: )' does not match and the pattern misses it. Additionally, the alternation anchor '^' applies only to the very start of the line; combined with the '-E' flag the full pattern '(^|: )(warning|error):' correctly fires on line-start OR colon-space — but a message like '/src/foo.hpp:10: warning: ...' has ': warning:' after the line number WITH a space? Let me re-examine: ':10: warning:' — the text immediately before 'warning:' is ': ', so the colon-space branch DOES match here. The real gap is messages that begin with 'warning:' without a colon-space prefix on a line that also has other text first; this is uncommon in practice but untested. The filter patterns do not require a test suite, so the only assurance is a manual read.
   *Recommendation:* Add a pair of inline unit tests (e.g. a bats or simple bash heredoc test) that feed a synthetic log containing a known doxygen warning line and a known filtered line through the grep pipeline, asserting the right lines are kept/dropped. This would also protect the filter regex from silent breakage on grep version differences (BSD vs GNU -E behaviour).
   *Fixed:* Added representative example comments for each filter arm (E1-E6) so reviewers can verify the pattern without running doxygen.
