8. [ ] **performance-reviewer** | `scripts/check-parallel-install.sh:213` | missing-caching
   The v1 (master) build — bootstrap, configure, make -j$(nproc) — runs from scratch on every CI invocation with no artifact caching. The EXIT trap removes $MASTER_WORKTREE unconditionally, so nothing is reused across runs. On a GitHub Actions ubuntu-latest runner (2 cores) a full libhttpserver autotools build adds roughly 3–6 minutes to the gcc-14 extra lane on every PR. This is acceptable because the gate is correctly gated to a single matrix lane (parallel-install: check), but a cache keyed on the v1 master commit SHA could reduce this to near-zero on non-force-push runs.
   *Recommendation:* Add an actions/cache step before the parallel-install gate that caches $BUILD_DIR/.parallel-stage (or a separate v1 install prefix) keyed on ${{ hashFiles('') }} of the master ref tip SHA. On a cache hit the gate can skip Phase 2 and jump straight to Phase 3 (coexistence assertion). Alternatively, cache only the installed v1 artifacts (the DESTDIR subtree) rather than the full build tree to keep the cache entry small.

9. [ ] **security-reviewer** | `scripts/check-parallel-install-lane.sh:116` | insecure-design
   The structural gate that prevents the HTTPSERVER_ALLOW_PARALLEL_INSTALL_SKIP escape hatch from being set in CI (assertion (e)) lives in the same repository as the workflow file it guards. A PR author could modify both check-parallel-install-lane.sh (removing assertion (e)) and .github/workflows/verify-build.yml (adding the escape hatch) in a single PR, bypassing the automated safeguard entirely. Since pull_request workflows execute the PR branch's copy of the workflow and the gate script, no independent trusted copy exists to catch this. The defence is therefore limited to human code review of the PR diff.
   *Recommendation:* This is a known structural limitation of self-hosted CI gate scripts; it cannot be fully closed without an externally-hosted policy layer (e.g. a branch protection rule enforcing required status checks that themselves are not modifiable via PR). Document this limitation as a review requirement in CONTRIBUTING.md or the gate's header comment: reviewers must verify that neither check-parallel-install-lane.sh nor the parallel-install workflow steps were modified to weaken assertion (e).

10. [ ] **security-reviewer** | `scripts/check-parallel-install.sh:161` | injection
   CWE-88 (Argument Injection): $MASTER_REF is passed to `git rev-parse --verify "$MASTER_REF"` and `git worktree add --detach "$MASTER_WORKTREE" "$MASTER_REF"` without an end-of-options `--` separator. If MASTER_REF were set to a value beginning with `--` (e.g. MASTER_REF=--short), git would interpret it as a flag rather than a refspec. In CI, MASTER_REF is hardcoded as `origin/master` via the preceding workflow step, so immediate exploitability is nil; the risk is limited to a developer intentionally or accidentally setting the variable to an option-like value in a local run.
   *Recommendation:* Add the end-of-options delimiter where the git command supports it: `git -C "$REPO_ROOT" rev-parse --verify -- "$MASTER_REF"`. For `git worktree add`, which does not accept `--` before the commit-ish argument in current git releases, add a pre-flight format check: `git check-ref-format --allow-onelevel "$MASTER_REF" || fail "MASTER_REF contains invalid characters: $MASTER_REF"`.

11. [ ] **security-reviewer** | `scripts/check-parallel-install.sh:255` | input-validation
   The fallback coexistence check parses `ls "$STAGE_LIB"` output via a pipe to `grep -E "$V1_PATTERN"` and `grep -v`. In the narrow case where a library filename contains an embedded newline, the ls-pipe approach could produce incorrect results (a filename split across lines matching neither pattern, or a line being mistaken for a different filename). V1_PATTERN is a hardcoded regex and STAGE_LIB is created by the controlled build process, so real-world exploitability is negligible.
   *Recommendation:* Replace the ls-pipe pattern with a glob-based check: `v1_hits=""` then loop `for f in "$STAGE_LIB"/libhttpserver.*; do [[ "$f" =~ $V1_PATTERN ]] && ...; done`. This is newline-safe and does not depend on ls output parsing.
