21. [ ] **security-reviewer** | `src/webserver.cpp:291` | injection
   classify_route_tier() compiles a std::regex from the normalized route pattern at registration time (std::regex::extended | std::regex::icase). Route patterns are typically supplied by the application developer at startup, not by end users, so the attack surface is narrow. However, if any downstream integration allows route patterns to be configured at runtime from external input (e.g., config files, admin APIs), a maliciously crafted pattern could trigger catastrophic backtracking (ReDoS, CWE-1333).    *Recommendation:* Add a size cap on registered route patterns (e.g., reject patterns longer than 512 characters) and consider wrapping the std::regex constructor in a try/catch for std::regex_error to surface malformed-pattern errors early rather than propagating a thrown exception up through register_path/register_prefix without context. If admin-controlled route registration is ever exposed, add a timeout-limited match guard or switch to a linear-time engine (e.g., RE2).
