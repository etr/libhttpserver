1. [x] **code-quality-reviewer** | `src/webserver.cpp:2097` | error-handling
   ws_upgrade_data is allocated with raw `new` at line 2097. If MHD_create_response_for_upgrade returns nullptr the allocation is freed at line 2113 via `delete data`, but the shared_ptr<websocket_handler> inside the struct is destroyed only if delete is reached. Between line 2097 and the nullptr-branch check at line 2099, any future exception path (e.g. if MHD_create_response_for_upgrade throws internally on some platform/version) would leak the allocation.    *Recommendation:* Use std::make_unique<ws_upgrade_data> and release() only on the success path, as is already done for the upgrade_handler's incoming cls (line 1654). This closes the same gap that was present in the v1 raw-handler path and keeps the ownership model consistent throughout.
   **Fixed:** Already addressed in the merged code (complete_websocket_upgrade uses unique_ptr data_guard). Further tightened data_guard.release() to only fire on MHD_queue_response success (finding 14) and Sec-WebSocket-Accept failure now aborts the handshake (finding 15).

14. [x] **security-reviewer** | `src/webserver.cpp:2097` | insecure-design
   ws_upgrade_data is heap-allocated with `new` at line 2097, then passed to MHD_create_response_for_upgrade.    **Fixed:** data_guard.release() now only fires when MHD_queue_response returns MHD_YES; otherwise the destructor frees the allocation.
