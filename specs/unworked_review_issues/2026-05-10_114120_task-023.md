4. [ ] **performance-reviewer** | `src/webserver.cpp:1427` | race-condition
   The LRU route cache is guarded by a plain std::mutex (route_cache_mutex) while the route table itself uses a std::shared_mutex. On every cache hit the dispatch thread must acquire an exclusive lock on route_cache_mutex, serialising all threads at exactly the hottest point of the lookup path. The cache was introduced to short-circuit the O(n) regex scan; its own lock now becomes the bottleneck under concurrent load because it cannot be shared across readers.
   *Recommendation:* Separate read-only cache probes from write (LRU promotion + insert). One option: use a read/write split where a cache hit only promotes under a try-lock and skips promotion on contention (acceptable for an approximate LRU). Another option: replace the LRU std::list+std::unordered_map with a lock-free structure (e.g. a fixed-size, epoch-based concurrent map) or an RCU snapshot. At minimum, consider making route_cache_mutex a std::shared_mutex so multiple threads can read the cache simultaneously, and only upgrade to exclusive for promotion/insert.
