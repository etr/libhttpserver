### DR-013: Digest auth simplified to static WWW-Authenticate challenge

**Status:** Accepted
**Date:** 2026-05-04
**Context:** v1's `basic_auth_fail_response` and `digest_auth_fail_response` each produced their 401 challenges via dedicated response subclasses; `digest_auth_fail_response` in particular delegated to `MHD_queue_auth_required_response3`, which sets the full RFC 7616 nonce, opaque, algorithm, and qop fields and wires the challenge into libmicrohttpd's per-connection Digest state machine. v2.0 removes all `*_response` subclasses and the `get_raw_response` / `decorate_response` / `enqueue_response` virtuals in favour of the unified `http_response::unauthorized(scheme, realm, body)` factory. The factory returns an `http_response` by value with no MHD connection handle available, so it cannot call `MHD_queue_auth_required_response3` — that API requires a live connection pointer, which is incompatible with the value-typed response model.

**Options considered:**
1. **Static challenge only.** `unauthorized("Digest", realm)` emits `WWW-Authenticate: Digest realm="<realm>"` without nonce/opaque/qop. Simple, consistent with the value-type model. Non-RFC-compliant: RFC 7616 §3.3 requires a nonce in every 401 Digest challenge. Clients performing strict RFC 7616 parsing may reject the challenge or fail to fall back.
2. **Reject `"Digest"` scheme at the factory with `std::invalid_argument`.** Forces callers to use MHD APIs directly for Digest auth. Maximally safe, but breaks any caller that currently uses the factory for a Digest stub.
3. **Keep a separate `MHD_queue_auth_required_response3`-based code path triggered by scheme == "Digest".** Requires threading a connection handle through the response value, breaking the DR-005 value-type model.
4. **Defer Digest support to a future task using a separate API surface** (e.g. a `webserver`-side hook or connection-time callback). Zero breakage today; full RFC compliance later.

**Decision:** Option 1 for v2.0, with Option 4 reserved for a future v2.x task.

**Rationale:** Options 3 and 4 both require API surface that was not scoped into TASK-013. Option 2 is preferred from a safety standpoint but would break the existing test suite that exercises the Digest static-challenge path (PRD §3.5 acceptance tests). Option 1 preserves backwards compatibility of the factory surface while making the limitation visible via a `@warning` Doxygen tag on `unauthorized()` and a note in the architecture doc. The existing test fixtures (TASK-013 §2/§10) document the expected FAIL body, and the static challenge is sufficient for callers that only need a 401 with a WWW-Authenticate header (e.g. tests, basic HTTP browsers that fall back to Basic).

**Consequences:**
- `unauthorized("Digest", realm)` produces a non-RFC-7616-compliant challenge: no `nonce`, no `opaque`, no `algorithm`, no `qop`.
- Callers needing full RFC 7616 Digest auth (nonce handshake, replay protection) must use `MHD_queue_auth_required_response3` directly.
- The `@warning` Doxygen tag on `unauthorized()` in `http_response.hpp` makes the limitation visible at the API level (addressing PRD §3.5 review finding #1).
- Future v2.x may add a separate `webserver::require_digest_auth(resource, realm)` API that operates at connection time and can invoke the MHD APIs correctly.

**Related:** PRD-RSP-REQ-005, TASK-013 §2/§10, http-response.md §4.3.
