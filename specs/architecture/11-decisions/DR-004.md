### DR-004: Handler return type

**Status:** Accepted
**Date:** 2026-04-30
**Context:** PRD originally said handlers return `unique_ptr<http_response>` while factories return by value — internal contradiction. With DR-3a making `http_response` a value type, return-by-value is cheap and natural.

**Options considered:**
1. **Return `http_response` by value** — handler signature `http_response(const http_request&)`; matches Crow.
2. **Return `unique_ptr<http_response>`** (original PRD) — explicit ownership; forces a heap allocation we just removed in DR-3a.
3. **Return `std::optional<http_response>`** — `nullopt` means fallthrough; we don't have handler chains in v2.0 (YAGNI).

**Decision:** Option 1.

**Rationale:** With value semantics, wrapping in `unique_ptr` adds ceremony for no benefit. Return-by-value lets the factory chain BE the return statement: `return http_response::string("ok").with_status(201);`. Delivers the PRD's "≤10 LOC hello world" JTBD literally. Option 3 solves a problem (handler chaining) we don't have.

**Consequences:**
- PRD-HDL-REQ-001 amended to require `std::function<http_response(const http_request&)>`.
- PRD-RSP-REQ-007 amended to require `http_response` by value.
- No handler null-pointer ambiguity (a returned `http_response` is always valid).
