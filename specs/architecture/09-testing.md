## 9) Testing strategy

The architecture itself does not prescribe test frameworks (out of architecture scope), but it does name the test surfaces that need first-class coverage given v2.0's structural changes:

1. **Header hygiene** (PRD-HDR-REQ-001..003): a CI test compiles a TU containing only `#include <httpserver.hpp>` and `int main() {}` with no `-I` to libmicrohttpd / pthread / gnutls headers.
2. **Build-flag invariance** (PRD-FLG-REQ-001): the same consumer source compiles against `--disable-tls` and `--enable-tls` builds without changes.
3. **Move semantics on `http_response`** (DR-5): sanitizer-clean tests for inline↔inline, inline↔heap, heap↔inline, heap↔heap on both move-construct and move-assign. The sanitizer matrix covers asan, lsan, tsan, ubsan, and msan.
   - Note: MemorySanitizer (msan) coverage is scoped to the backend-free unit-test subset (`unit/` targets in `test/Makefile.am`) because curl — used by all integration round-trips via `test/integ/curl_helpers.hpp` — cannot be practically instrumented; uninstrumented curl output would surface as false `use-of-uninitialized-value` reports. libc++ and libmicrohttpd are built instrumented in-CI (TASK-087) to keep the unit subset clean without wildcard suppressions.
4. **SBO size invariant** (DR-5): `static_assert(sizeof(detail::deferred_body) <= http_response::body_buf_size, ...)` at the end of `detail/body.hpp`. Compile-time guarantee.
5. **Routing semantics preservation** (DR-7): the v1 routing-test corpus runs against v2.0 unchanged. Any regression is treated as a release-blocker.
6. **Thread-safety contract** (DR-8): a stress test exercises `register_path` / `unregister_path` / `block_ip` / `unblock_ip` from within handlers, verifies no deadlock except for the documented `stop()` case. An opt-in negative test (`stop_from_handler_deadlocks_as_documented`, enabled via `HTTPSERVER_RUN_STOP_FROM_HANDLER=1`) confirms the deadlock contract by forking a child that calls `stop()` from inside a handler. The same stress test also exercises the server-wide hook bus (TASK-052) and the per-resource hook-bus CAS path (TASK-094 Sub-test D) under TSan.
7. **Performance acceptance gates** (PRD §3.6 / TASK-039 / TASK-083): `make bench` builds the five `bench_targets` in `test/Makefile.am`, each with its own pass/fail gate:
   - `bench_get_headers`: `get_headers()` warm-path cost ≥10× faster than v1 (PRD-REQ-REQ-001), checked at runtime.
   - `bench_sizeof_http_resource`: `sizeof(http_resource)` shrinks by at least the empty `std::map<std::string,bool>` footprint (PRD-REQ-REQ-003), checked at compile time.
   - `bench_hook_overhead` (TASK-052): the zero/one-hook dispatch overhead must stay within `2× HOOK_BASELINE_NS` (baseline measured in the same run) and under an absolute 50 ns sanity ceiling.
   - `bench_route_lookup` (TASK-053): two named measurements, each with its own ceiling — `cache_warm_ns` (cache hit) ≤ 200 ns, `radix_pure_ns` (cache cold, radix-only) ≤ 5 µs.
   - `bench_warm_path` (TASK-058): per-request allocation cost must not regress ≥5% versus a versioned, per-platform `BASELINE_NS` baseline.
   These benchmarks live in `EXTRA_PROGRAMS` (not `check_PROGRAMS`) so they do not run under sanitizer CI; they are gated on a quiet release-mode host as part of the release runbook.
8. **Valgrind race detectors — helgrind + drd** (DR-8 / TASK-088): the Linux valgrind CI lane runs memcheck, helgrind, and drd as three parallel GitHub Actions matrix entries (`build-type: valgrind-memcheck`, `valgrind-helgrind`, `valgrind-drd`), each invoking `make check-valgrind-<tool>` independently. The full `check_PROGRAMS` suite runs under all three tools; helgrind/drd scope is deliberately not narrowed to `unit/` tests only because the concurrency tests (`threadsafety_stress`, `route_table_concurrency`) are the primary reason for running race detectors (DR-008) and must not be subsetted out. Finding triage happens on the PR's Linux lane (Valgrind is Linux-only and cannot run on darwin); confirmed third-party benign races go in `test/valgrind-helgrind.supp` and `test/valgrind-drd.supp`. No wildcards spanning libhttpserver's own symbols are permitted (DR-008, mirroring `test/tsan.supp`). sgcheck (`exp-sgcheck`) is intentionally disabled: it was removed upstream in Valgrind 3.20 and ubuntu-latest ships 3.22 (`AX_VALGRIND_DFLT([sgcheck],[off])` in `configure.ac`). MemorySanitizer (msan) and Valgrind memcheck play complementary roles: msan runs the `unit/` subset with uninstrumented-library awareness (TASK-087); memcheck runs the full suite for leak/corruption detection where instrumented dependencies are not required.
9. **Shared integration-test helper convention** (TASK-075): `test/integ/` tests share `test_utils.hpp`, `curl_helpers.hpp`, and `server_ready.hpp` rather than re-implementing curl round-trips and server-ready waits per file. The `lint-server-ready-helper` gate (`scripts/check-server-ready-helper.sh`, wired into `check-local` via the top-level `Makefile.am`) forbids a bare `std::this_thread::sleep_for` server-ready wait in `test/integ/hooks_*.cpp` files unless the line carries a `// NON-READINESS-SLEEP: <reason>` marker.
10. **SKIP primitive and TLS-CLI-absent lane** (TASK-076): `LT_SKIP()` / `LT_SKIP_IF(cond, msg)` are a third outcome distinct from WARN/CHECK/ASSERT — they halt the test body without recording a failure, and a binary where every test SKIPs exits 77 so Automake reports it as skipped rather than failed. The `tls-no-cli` CI lane (`verify-build.yml`) runs the suite with `gnutls-cli` absent from `PATH` and asserts the five `psk_connection_*` tests report `[SKIP]` rather than `[PASS]`.
