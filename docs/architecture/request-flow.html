How a request flows — from the MHD callback to the wire

One HTTP request is not one function call. libmicrohttpd drives the exchange through a fixed sequence of C-ABI callbacks (the purple webserver_impl trampolines), each forwarding into a behavior service. This traces that journey end-to-end: the callback spine, the body-accumulation loop, the WebSocket branch, the four-tier route resolution, and where each of the 11 hook phases fires. Route resolution — route_table::lookup_v2

Step 15 above. Four tiers, cheapest first; the first hit wins and returns a lookup_result{found, tier, entry, captured_params}. Lookup keys off the raw standardized_url string — http_endpoint parsing is a registration-time concern, not per-request. The entry is returned regardless of method so the 405 path still sees it.

Method → handler. lookup_v2 never filters by method. The verb was already turned into a pointer-to-member (render_get/render_post/…) back in resolve_method_callback (step 8); dispatch_resource_handler then checks http_resource::is_allowed(method_enum) — mismatch yields 405 with the resource's Allow: header.

The WebSocket branch — diverges at step 13

Before any routing or handler dispatch, finalize_answer offers the request to the upgrader. A genuine upgrade queues 101 Switching Protocols and bypasses routing, all handler hooks, and the response materializer entirely.

The 11 hook phases, in firing order

Server-wide hooks live on hook_bus; five phases are also per-resource (resource_hook_table, registered via http_resource::add_hook). hook_dispatcher fires both chains. Every phase is guarded by a relaxed-atomic has_hooks_for check — zero cost when unused. Short-circuit phases can return respond_with(...) to end the request early.

Reading the flow. Purple bands are the fixed libmicrohttpd callback sequence — webserver_impl's static C-ABI trampolines, the only code touching MHD's ABI. Everything real happens in the teal behavior services they delegate to. The three branch axes that reshape the path: has_body (whether the body loop runs), skip_handler (a request_received/body_chunk short-circuit that skips routing straight to materialize), and the WebSocket upgrade (which bypasses the HTTP tail entirely). One request drives answer_to_connection 1..N times — once for a bodyless GET, many times while a POST body streams in — but resolves to exactly one finalize_answer.
